## 1. 当前技术栈分析

### 1.1 已识别技术栈

| 层级 | 技术 | 版本 | 当前用途 |
|------|------|------|----------|
| 框架 | Next.js App Router | 16.2.7 | 全栈框架，Route Handler 承载后端 |
| UI | React | 19.2.4 | 视图层 |
| 组件库 | Ant Design | 6.4.3 | Button、Empty、Layout 等基础组件 |

| Markdown | react-markdown + remark-gfm + react-syntax-highlighter | 10/4/16 | AI 回复的 Markdown 渲染和代码高亮 |
| ID | uuid | 14.x | `generateId()` 生成消息/会话唯一 ID |
| 状态管理 | Zustand | 5.0.14 | V5 正式启用（已安装），替代自定义 Hook 方案 |
| 持久化 | localStorage | — | key `chat_sessions` 存储全部会话数据 |
| 大模型 | OpenAI 兼容协议 SSE | — | `route.ts` 透传流式响应 |
| 错误处理 | 结构化 ChatErrorType | — | 8 种错误类型 + 重试机制 |

### 1.2 现有代码规范特征

| 特征 | 说明 |
|------|------|
| 状态管理 | Zustand store（V5 重构），`create` + `persist` 中间件，`get()` 读最新状态 |
| 组件模式 | 纯展示组件（Props 接口驱动），逻辑集中在 Hook |
| 样式方案 | 全局 CSS 文件 `chatStyles.css`，class 命名扁平化（`chat-xxx`、`session-xxx`、`message-xxx`） |
| 类型定义 | 集中在 `chatTypes.ts`，interface 导出 + 工具函数导出 |
| 后端模式 | 单一 Route Handler (`POST /api/chat`)，直接代理大模型 SSE |

| 包名 | 用途 | 安装命令 |
|------|------|----------|

## 3. 核心模块划分及职责

### 3.1 模块依赖关系

```
layout.tsx (ConfigProvider + ThemeProvider)
    │
    └── chat/page.tsx (页面组装层)
         │
         ├── Zustand Stores（状态层）
         │    ├── useChatStore ──────────── sessions、消息收发、流式处理、重试、persist
         │    ├── useTemplateStore ────── 模板 CRUD、搜索、persist
         │    ├── useFavoriteStore ────── 收藏管理、persist
         │    └── useThemeStore ────────── 主题偏好、persist
         │
         ├── 计算型 Hooks（派生层）
         │    ├── useSessionManager ──── 重命名/置顶/归档/排序/批量（读 useChatStore）
         │    ├── useSearch ────────────── 全局搜索（读 useChatStore.sessions）
         │    └── useExport ───────────── 导出（读 useChatStore.sessions）
         │
         └── 子组件可直接 useChatStore(selector) 订阅所需字段
```

### 3.2 各模块职责边界

| 模块 | 类型 | 职责 | 持久化 | 数据源 |
|------|------|------|--------|--------|
| `useChatStore` | Zustand Store | sessions 数据源、消息发送、流式接收、重试、停止生成、mode 切换 | `persist` 中间件 → `chat_sessions` | 唯一数据源 |
| `useTemplateStore` | Zustand Store | 内置+自建模板列表、CRUD、搜索过滤、使用计数 | `persist` 中间件 → `custom_templates` | 独立 |
| `useFavoriteStore` | Zustand Store | 收藏/取消收藏、收藏列表管理、搜索 | `persist` 中间件 → `chat_favorites` | 独立，通过 `get()` 读 chatStore |
| `useThemeStore` | Zustand Store | 主题偏好、系统主题监听、`data-theme` 切换 | `persist` 中间件 → `theme_preference` | 独立 |
| `useSessionManager` | 计算型 Hook | 会话列表 UI 状态：重命名、置顶、归档、搜索过滤、拖拽排序、批量 | 无（写回 chatStore） | 订阅 chatStore |
| `useSearch` | 计算型 Hook | 全局消息搜索、结果高亮、跳转定位 | 无（内存计算） | 订阅 chatStore.sessions |
| `useExport` | 计算型 Hook | JSON/Markdown 导出、ZIP 打包、触发下载 | 无 | 订阅 chatStore.sessions |

### 3.3 关键设计决策

**为什么 V5 启用 Zustand 替代自定义 Hook？**

| 痛点 | 当前代码现状 | Zustand 解决方式 |
|------|--------------|--------------------|
| useRef 同步胶水 | 3 个 `useRef` + 3 个 `useEffect` 专门同步状态给异步回调（12行代码） | `get()` 直接读最新值，彻底消除 |
| V5 prop drilling | 6+ 个新组件（GlobalSearch、FavoritesDrawer、ExportMenu 等）都需要 sessions，只能从 page.tsx 逐层传递 | 任意组件 `useChatStore(s => s.sessions)` 直接订阅 |
| useCallback 膨胀 | 每个 action 都需要 `useCallback` + 依赖数组（~50行胶水） | action 直接定义，无需 useCallback |
| localStorage 持久化 | 手写 `useEffect` 监听 sessions 变化 + `setItem`（8行） | `persist` 中间件一行配置 |
| 精确渲染 | Hook 返回大对象，任意字段变化都触发所有消费者 re-render | selector 精确订阅，只有关心字段变化才 re-render |

重构风险评级：🟢 **低风险**（仅 2 个文件需改动：`useChatStore.ts` 重写 + `page.tsx` 小改，子组件零改动）

**Store 分层策略：为什么拆成多个 Store 而不是一个大的？**

- `useChatStore`：聊天核心数据（sessions/messages）+ 流式请求逻辑，是最频繁使用的 store
- `useTemplateStore`：模板数据独立，与聊天无耦合，单独 persist
- `useFavoriteStore`：收藏数据独立，需要跨 store 读取 sessions 时用 `useChatStore.getState()`
- `useThemeStore`：主题偏好完全独立，与业务数据无关
- 多个小 store 比一个大 store 更好：职责清晰、persist key 分离、渲染隔离

**计算型 Hook vs Zustand Store 的边界？**

- **用 Zustand Store**：需要持久化到 localStorage 的数据（sessions、templates、favorites、theme）
- **用计算型 Hook**：纯派生计算，不需要持久化（搜索、导出、会话管理 UI 状态）
- `useSessionManager` 是计算型 Hook，它订阅 `useChatStore` 的 sessions，提供列表 UI 操作（重命名/置顶/排序/批量），操作结果写回 chatStore

### 4.2 useChatStore Zustand Store（重构后核心结构）

```ts
import { create } from 'zustand'
import { persist } from 'zustand/middleware'

interface ChatStoreState {
  // === State ===
  sessions: ChatSession[]
  currentSessionId: string | null
  currentMode: ChatMode | null
  loading: boolean

  // === Actions ===
  sendMessage: (content: string) => Promise<void>
  stopGeneration: () => void
  retryMessage: (robotMessageId: string) => Promise<void>
  createNewSession: () => string
  switchSession: (sessionId: string) => void
  deleteSession: (sessionId: string) => void
  clearCurrentSession: () => void
  setMode: (mode: ChatMode | null) => void
  // V5 新增：外部写入 sessions（供 useSessionManager 调用）
  updateSessions: (updater: (prev: ChatSession[]) => ChatSession[]) => void
}

export const useChatStore = create<ChatStoreState>()(
  persist(
    (set, get) => ({
      sessions: [],
      currentSessionId: null,
      currentMode: null,
      loading: false,

      // 不再需要 useRef 同步，直接 get() 读最新值
      sendMessage: async (content: string) => {
        const { sessions, currentSessionId, currentMode, loading } = get()
        if (!content.trim() || loading) return
        // ... 流式请求逻辑不变，内部用 set() 更新状态
      },

      updateSessions: (updater) =>
        set(state => ({ sessions: updater(state.sessions) })),

      // ... 其他 action 同理
    }),
    { name: 'chat_sessions' }  // persist 一行搞定
  )
)

// 子组件精确订阅示例：
// const sessions = useChatStore(s => s.sessions)
// const loading = useChatStore(s => s.loading)
```

## 5. 数据流转逻辑

### 5.1 消息发送完整链路（现有 + V5 扩展）

```
用户输入 → ChatWindow.handleSend()
  → useChatStore.sendMessage(content)
    → 构建 contextMessages
    → POST /api/chat { messages, mode }
      → route.ts: 截断上下文(20条) → 构建 systemPrompt → 调大模型 SSE
    → 流式读取 → updateRobotMessage(累积)
    → 流结束
      → ✅ 第一轮对话？→ 异步 POST /api/generate-title → 更新会话标题
      → ✅ 用户手动重命名过？→ 跳过标题生成
```

### 5.2 会话管理数据流

```
用户操作 SessionList
  → useSessionManager 方法
    → useChatStore.getState() 读最新 sessions
    → useChatStore.updateSessions() 写回
    → zustand persist 中间件自动同步 localStorage
    → selector 订阅者自动 re-render

拖拽排序：
  SessionItem onDragEnd
    → useSessionManager.reorderSessions(from, to)
    → 更新每个 session.sortOrder
    → zustand persist 自动持久化

批量删除：
  Ctrl+Click 选中 → selectedIds.add()
  点击「批量删除」→ batchDelete()
    → useChatStore.getState().deleteSession(id) 逐个删除
    → exitBatchMode()
```

### 5.3 模板使用数据流

```
用户打开 TemplatePanel
  → useTemplates 返回 filteredTemplates
用户点击模板卡片
  → useTemplates.useTemplate(id) → 返回 systemPrompt，usageCount++
  → 页面层：
    → createNewSession()
    → setMode(null)  // 清除学习模式
    → 将 systemPrompt 注入（扩展 ChatSession 增加 customSystemPrompt 字段）
    → 显示模板专属欢迎页
```

### 5.4 收藏数据流

```
用户点击消息气泡 ⭐ 按钮

    → 创建 FavoriteItem 快照

  → 切换到对应会话 + 滚动到消息位置
```

### 5.5 导出数据流

```
用户点击导出
  → useExport.exportSession(id, format)
    ├── format='json'  → sessionToJson() → Blob → 下载 .json

### 5.6 主题切换数据流

```

  → useTheme()

## 6. 技术风险及应对方案

### 6.1 风险矩阵

| 风险 | 等级 | 影响 | 应对方案 |
|------|------|------|----------|
| **localStorage 容量溢出** | 🔴 高 | 会话数据 + 模板 + 收藏可能超过 5MB 限制 | 见 6.2 |
| **zustand 重构回归风险** | 🟡 中 | 流式 SSE 逻辑从 useCallback 迁移为 zustand action，写法变化可能引入 bug | 见 6.3 |
| **暗黑模式颜色遗漏** | 🟡 中 | chatStyles.css 有 900+ 行，可能有硬编码颜色未替换 | 见 6.4 |
| **@dnd-kit 与 Ant Design Sider 冲突** | 🟡 中 | 拖拽事件可能被 Sider 的 overflow 截断 | 见 6.5 |
| **标题生成额外 API 调用成本** | 🟢 低 | 每次新会话多一次大模型调用 | 见 6.6 |
| **jszip 包体积** | 🟢 低 | 增加约 100KB gzipped | 动态 import，仅导出时加载 |

### 6.2 localStorage 容量溢出应对

**问题**：localStorage 限制 5MB，大量会话 + 长消息 + 收藏快照可能接近上限。

**方案**：

```ts
// lib/storage.ts
export function safeSetItem(key: string, value: string): boolean {
  try {
    localStorage.setItem(key, value);
    return true;
  } catch (e) {
    if (e instanceof DOMException && e.name === 'QuotaExceededError') {
      console.warn(`localStorage 写入失败: ${key}，尝试清理旧数据`);
      // 策略：删除最旧的 20% 会话消息内容（保留会话结构）
      return cleanupAndRetry(key, value);
    }
    return false;
  }
}

function cleanupAndRetry(key: string, value: string): boolean {
  if (key === STORAGE_KEYS.SESSIONS) {
    const sessions = JSON.parse(value) as ChatSession[];
    // 按 updatedAt 排序，清理最旧 20% 会话的消息内容
    const sorted = [...sessions].sort((a, b) => a.updatedAt - b.updatedAt);
    const cutoff = Math.ceil(sorted.length * 0.2);
    const ids = new Set(sorted.slice(0, cutoff).map(s => s.id));
    const cleaned = sessions.map(s =>
      ids.has(s.id) ? { ...s, messages: s.messages.slice(-5) } : s
    );
    try {
      localStorage.setItem(key, JSON.stringify(cleaned));
      return true;
    } catch {
      return false;
    }
  }
  return false;
}
```

**额外措施**：在 `useChatStore` 的 sessions 变化监听中加入容量检测，当使用率 > 80% 时提示用户导出并清理。

### 6.3 Zustand 重构回归风险

**问题**：流式 SSE 逻辑从 `useCallback` + `useRef` 迁移为 zustand action，写法变化可能引入 bug。

**方案**：

1. **分步迁移**：先重构数据定义和 CRUD action，验证通过后再迁移流式请求逻辑
2. **abortRef 保留**：`AbortController` 不是 React 状态，仍用 `useRef` 管理（在 store 外部）
3. **对照测试**：迁移后执行完整的浏览器验证流程（发送消息、流式输出、停止生成、重试、错误卡片）
4. **回滚预案**：git 分支隔离，出问题可随时回退到自定义 Hook 版本

### 6.4 暗黑模式颜色遗漏检测

**方案**：

1. 用脚本扫描 `chatStyles.css` 中所有硬编码颜色值：
   ```bash
   grep -nE '#[0-9a-fA-F]{3,8}|rgb\(|rgba\(' chatStyles.css
   ```
2. 逐个替换为 CSS Variable
3. 在 `theme.css` 中定义完整的 `[data-theme="dark"]` 变量集
4. 对 Ant Design 组件通过 `ConfigProvider` 的 `theme.token` 覆盖残余颜色

### 6.5 拖拽与 Sider 布局冲突

**方案**：

- `@dnd-kit` 的 `DndContext` 放在 `session-list-items` 容器内部，而非 Sider 层级
- 拖拽时设置 `overflow: visible` 临时覆盖 Sider 的 `overflow: hidden`
- 若仍有问题，降级为「上移/下移」按钮代替拖拽

### 6.6 标题生成成本控制

**方案**：

- 仅在**第一轮对话完成后**触发，后续对话不重复调用
- `assistantReply` 截取前 200 字即可，不传完整回复
- 使用最便宜的模型（如 `glm-4-flash`，已在 `.env` 中配置）
- 失败静默降级，不重试，保留截断标题
- `titleGenerated` 标记避免重复生成

## 7. 页面组件组合方式（chat/page.tsx 改造后）

```tsx
'use client';

import { ConfigProvider, theme as antTheme, Layout } from 'antd';
import { useChatStore } from '@/app/store/useChatStore';
import { useThemeStore } from '@/app/stores/useThemeStore';
import { useSessionManager } from '@/app/hooks/useSessionManager';

export default function ChatPage() {
  // Zustand stores —— 子组件也可以直接订阅，无需从这里传递
  const resolvedTheme = useThemeStore(s => s.resolvedTheme);
  const sessionManager = useSessionManager();  // 内部订阅 useChatStore

  // Ant Design 主题联动
  const algorithm = resolvedTheme === 'dark'
    ? antTheme.darkAlgorithm
    : antTheme.defaultAlgorithm;

  return (
    <ConfigProvider theme={{ algorithm }}>
      <Layout className="chatbot-page">
        {/* Sider: SessionList、GlobalSearch 等组件内部直接 useChatStore(selector) */}
        {/* Content: ChatWindow、ModeSelector、TemplatePanel 等 */}
      </Layout>

```

> **关键变化**：page.tsx 从“数据分发中心”简化为“布局组装层”。子组件通过 zustand selector 自行订阅所需数据，不再依赖 props 逐层传递。
