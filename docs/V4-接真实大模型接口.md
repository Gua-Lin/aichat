# V4：接入真实大模型接口 — 需求规格说明

> **前置条件**：V1–V3 的 UI（会话列表、聊天气泡、学习模式选择器、Markdown 渲染等）已稳定运行。

---

## 1. 目标

将模拟回复替换为真实大模型流式回复，同时保持现有 UI 交互不变。API Key 仅存于服务端，前端不暴露任何密钥。

---

## 2. 技术选型

| 层级 | 技术 | 说明 |
|------|------|------|
| 框架 | Next.js App Router | 使用 Route Handler (`app/api/chat/route.ts`) 承载后端逻辑 |
| 语言 | TypeScript | 全栈类型安全 |
| UI | React + Ant Design + Tailwind CSS | 沿用现有组件体系 |
| 状态管理 | Zustand / 自定义 Hook | 沿用 `useChatStore` |
| 大模型接口 | OpenAI 兼容协议（SSE） | 统一对接 `/chat/completions` 端点，兼容 OpenAI、智谱、DeepSeek 等 |
| 流式传输 | 原生 `fetch` + `ReadableStream` | 手动解析 SSE，不引入 Vercel AI SDK 等额外依赖 |

> **为什么不用 Vercel AI SDK？**  
> 手动实现 SSE 解析能更好地理解流式通信原理，且项目已有完整的 `ReadableStream` 读取逻辑，无需额外依赖。如果后续需要简化，可考虑引入 `ai` 包的 `useChat`。

---

## 3. 环境变量配置

在 `.env.local` 中配置（参考 `.env.example`）：

```env
# 必填：大模型 API Key
OPENAI_API_KEY=sk-your-api-key-here

# 可选：API 基地址（默认 https://api.openai.com/v1）
# 智谱 AI：https://open.bigmodel.cn/api/paas/v4/
# DeepSeek：https://api.deepseek.com/v1
OPENAI_BASE_URL=https://open.bigmodel.cn/api/paas/v4/

# 可选：模型名称（默认 gpt-3.5-turbo）
# 智谱 AI：glm-4-flash
# DeepSeek：deepseek-chat
OPENAI_MODEL=glm-4-flash
```

## 4. 后端接口设计

### 4.1 端点

```
POST /api/chat
```

### 4.2 请求体

```ts
interface ChatRequest {
  messages: { role: 'user' | 'assistant'; content: string }[];
  mode: ChatMode | null;  // 'explain' | 'interview' | 'codeReview' | 'plan' | 'summary'
}
```

### 4.3 响应

- **成功**：返回 `text/event-stream`，格式遵循 OpenAI SSE 协议

  ```
  data: {"id":"...","choices":[{"delta":{"content":"你"},"index":0}]}

  data: {"id":"...","choices":[{"delta":{"content":"好"},"index":0}]}

  data: [DONE]
  ```

- **失败**：返回 JSON 错误信息

  | 状态码 | 场景 |
  |--------|------|
  | 400 | 请求体格式错误 / messages 为空 |
  | 502 | 大模型 API 返回错误 |
  | 503 | API Key 未配置 |
  | 500 | 服务端未知错误 |

### 4.4 System Prompt 策略

根据前端传入的 `mode` 字段动态切换系统提示词：

| 模式 | System Prompt 摘要 |
|------|---------------------|
| 无模式 | 通用中文 AI 助手 |
| explain | 耐心的老师，用通俗语言 + 比喻 + 例子解释概念 |
| interview | 资深前端面试官，问答式考察并追问细节 |
| codeReview | 严格的代码审查者，逐行分析并给出修正代码 |
| plan | 学习规划师，制定分步骤计划含里程碑和检验方式 |
| summary | 知识提炼专家，输出结构化要点总结 |

## 5. 前端流式接收

### 5.1 核心流程

```
用户发送消息
  → 构建上下文消息数组（历史消息 + 当前输入）
  → POST /api/chat（携带 mode）
  → 创建空的机器人消息占位
  → 逐块读取 SSE 流
  → 解析 data: JSON → 提取 delta.content
  → 累积拼接，实时更新消息内容
  → 流结束 / [DONE] → 完成
```

### 5.2 异常降级

```
API 请求失败
  ├── AbortError（用户主动停止）→ 保留已生成内容
  └── 其他错误 → 降级为本地 mockReply() 模拟回复
```

### 5.3 停止生成

- 使用 `AbortController` 控制请求生命周期
- 点击「停止生成」按钮调用 `controller.abort()`
- 已接收的流式内容保留在消息中，不丢失

## 7. 安全注意事项

1. **API Key 仅存于 `.env.local`**，通过 Route Handler 在服务端读取，前端永远不接触密钥
2. `.env.local` 已在 `.gitignore` 中，不会提交到代码仓库
3. Route Handler 对请求体做 try-catch 校验，防止畸形请求
4. 使用 `req.signal` 透传 AbortSignal，避免无效请求占用服务端资源
